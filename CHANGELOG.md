# Changelog

## [0.4.3] - 2026-05-19
### Fixed
- **`npm run fetch` hung after completion** (FIX 1): Anthropic SDK keeps HTTP keep-alive connections alive, preventing Node.js from exiting after `main()` resolved; added `.then(() => process.exit(0))` to `scripts/fetch.ts` and `scripts/brief.ts`; non-zero exits unchanged — `.catch` still calls `process.exit(1)`

### Changed
- **Haiku scoring prompt rewritten with niche priority stack** (CHG 1): replaced generic focus-areas list with explicit PRIMARY/SECONDARY/TERTIARY priority — Security (~50%: hacks, exploits, bridge attacks, smart contract vulnerabilities, audit findings, wallet security, phishing campaigns, bug bounties, formal verification), Tokenomics (~30%: token design, emission schedules, vesting cliff analysis, liquid staking (Lido, EtherFi, Rocket Pool), restaking (EigenLayer, Symbiotic), protocol revenue, treasury management, fee mechanisms, airdrop mechanism critique), L1/L2 Infrastructure (~20%: Ethereum upgrades + EIPs, L2 rollups (Base, Arbitrum, Optimism, zkSync, StarkNet), Bitcoin Optech/taproot/Lightning/OP_CAT, DA layers (EigenDA, Celestia), MEV/PBS/shared sequencing); scoring rubric updated to align with 7.5 HOT threshold — 9.0–10.0 major security incident or structural tokenomics change, 7.5–8.9 clear niche development with angle potential, 5.0–7.4 tangential, 3.0–4.9 general crypto, 0.0–2.9 spam/auto-dismissed; penalty areas and 12-category enum unchanged; existing scored articles not affected (scoring is idempotent)
- **`CLAUDE.md` release workflow section added** (CHG 2): embedded the 6-step release checklist (npm test → tsc --noEmit → version bump → CLAUDE.md → CHANGELOG → optional docs update) directly in `CLAUDE.md`

## [0.4.2] - 2026-05-16
### Fixed
- **`list --hot` appeared to ignore 7.5 threshold** (FIX 1): investigation confirmed the threshold was correctly `>= 7.5` in all queries since v0.4.1; the root cause of the observation was a data distribution issue — there are 82 articles scoring 8.0+ in the 30-day window, which fills `--limit=50` before reaching the 7.5–7.9 band (57 articles); the fix is a UX improvement rather than a logic fix

### Changed
- **List footer now shows total count when limit is hit** (CHG 1): `renderTieredView` footer changed from `"Showing 20 hot, 15 other"` to `"Showing 20 of 139 hot, 15 of 423 other (last 30 days)"` — the `"of N"` suffix appears only when the displayed count is less than total; `renderFlatView` footer similarly changed from `"Showing 50 article(s)"` to `"Showing 50 of 139 article(s)"`; footer hint updated from `"Use --days=N to change"` to `"Use --days=N or --limit=N to adjust"`
- **Refactored `list.ts` condition building** (CHG 2): extracted shared WHERE clause logic from `queryScoredArticles` into `buildArticleConditions(flags)` + `ARTICLE_JOINS` constant; added `countScoredArticles(flags, db)` reusing the same builder for COUNT(*) queries; no behavior change — same SQL conditions as before, just de-duplicated

## [0.4.1] - 2026-05-16
### Fixed
- **Recency filter used `fetched_at` as fallback** (FIX 1): `queryScoredArticles` filtered by `COALESCE(a.published_at, a.fetched_at) >= datetime('now', '-N days')` — articles with no `published_at` fell back to `fetched_at` (the DB insert time); articles re-inserted during the v0.3.0 source repair had a recent `fetched_at` despite being published in 2025, so old Optimism (Jun 2025) and Blockworks (Dec 2025) articles surfaced in the HOT list; fix: condition changed to `a.published_at >= datetime('now', '-N days')` — articles without a publication date are excluded from the default view (not a regression — no published_at means no date to filter on)
- **Haiku assigned free-form categories** (FIX 2): Haiku prompt had no category constraint; 429 scored articles → 200+ unique strings (`"DeFi"`, `"DeFi/Stablecoins"`, `"defi"`, `"DEFI Protocol"`, etc.), making `npm run stats:scoring` category breakdown useless; fix: system prompt appends a fixed enum (`L2 | DeFi | AI x Crypto | Developer Tooling | SocialFi | Bitcoin | Regulation | Macro | Research/Protocol | Price/Trading | Exchange/Corporate | Other`) and user prompt adds `IMPORTANT: "category" must be exactly one of: ...`; existing articles keep their free-form categories (no re-score); new articles from next `npm run fetch` will use the enum

### Changed
- **HOT threshold lowered 8.0 → 7.5** (CHG 1): `SCORE_THRESHOLDS.HOT` in `src/config/index.ts` changed from `8.0` to `7.5`; `npm run inspect:near-hot` had identified 57 articles in the 7.0–7.9 band that were substantive enough to surface; constant is centralized — all queries and display strings import from one place; scores in DB are unchanged, threshold only affects queries and display; histogram in `stats:scoring` now uses 8 buckets: replaces the single `7–8` bucket with `7–7.5` (near-HOT) and `7.5–8` (← HOT); files updated: `list.ts`, `content-filter/index.ts`, `haiku-filter.ts`, `stats-scoring.ts`, `stats-sources.ts`, `inspect-near-hot.ts`

## [0.4.0] - 2026-05-16
### Added
- **`npm run stats:scoring`** (ADD 1): `src/scripts/stats-scoring.ts` — read-only diagnostic for Haiku scoring distribution over the last N days (default 7, `--days=N` to override); prints total scored / HOT / OTHER / dismissed counts with percentages, avg score, score histogram with bar chart across 7 buckets (0–3, 3–5, 5–6, 6–7, 7–8 ← near-HOT, 8–9 ← HOT, 9–10 ← HOT), and Haiku-assigned category breakdown sorted by count; no API key required
- **`npm run stats:sources`** (ADD 2): `src/scripts/stats-sources.ts` — read-only per-source scoring breakdown over N days; prints aligned table with: source name, total scored articles, avg score, HOT count, OTHER count, dismissed count, HOT% (articles that scored ≥ 8 as share of total); rows ordered by HOT count then avg score descending; footer row shows cross-source totals; `✓` flag marks sources that produced at least one HOT article
- **`npm run inspect:near-hot`** (ADD 3): `src/scripts/inspect-near-hot.ts` — read-only; queries articles with `score >= 7.0 AND score < 8.0` in the last N days (default 30), shows up to `--limit=N` (default 10) results; per article prints full score, source name, published date, full title, full Haiku reasoning (untruncated), and suggested angle if present; useful for diagnosing whether HOT threshold is too strict

## [0.0.3] - 2026-04-16
### Added
- **Claude Haiku batch scoring** (ADD 1): `haiku-filter.ts` sends up to 10 articles per Anthropic API call with title + content snippet; response is a JSON array with `id`, `score` (decimal), `category`, `reasoning`, `suggestedAngle`; `scoreArticles()` clamps score to `[0, 10]`, skips malformed items with a warn log rather than throwing; batches run sequentially to avoid rate limits; cost tracked per call (input × $0.80/1M + output × $4.00/1M tokens) and surfaced in fetch summary
- **Decimal scores (0.0–10.0)** (ADD 2): Haiku prompt instructs model to use one decimal place (e.g. 7.2, 8.5) and avoid rounding to integers; scoring rubric: 9–10 significant protocol/research, 7–8.9 clear angle exists, 5–6.9 tangential, 3–4.9 mostly irrelevant, 0–2.9 spam/price prediction; focus areas scored higher (L2, DeFi, SocialFi, AI×Crypto, developer tooling); penalty areas scored lower (price analysis, celebrity endorsements, meme coins, exchange listings, listicles)
- **Tiered inbox** (ADD 3): `npm run list` shows HOT (score ≥ 8) first with suggested angle, then OTHER (6–7.9) without; `--hot` shows only HOT; `--other` shows only OTHER; `--all` shows everything including dismissed; recency filter defaults to 30 days (`--days=N` to override); `--today` convenience alias for `--days=1`; `--source=<slug>` and `--state=<state>` for additional filtering; score + article ID + source + time-ago on each line
- **Auto-dismiss below 6** (ADD 4): `filterNewArticles()` calls `updateArticleState(id, 'dismissed')` for any article scoring below 6 immediately after `insertFilterResult`; articles stay in DB (URL dedup still works) but are hidden from default list view; `--all` flag reveals them
- **Fallback score 5 on parse failure** (ADD 5): when Haiku returns malformed JSON or a non-array response, `buildFallbackResult()` assigns score 5 (`category: 'unknown'`, `reasoning: 'Fallback score — Haiku response could not be parsed'`) to all articles in the batch; articles land in OTHER tier rather than being lost; real parse errors logged at warn level
- **Idempotent scoring** (ADD 6): `isArticleScored()` checks `filter_results` before scoring; articles already in the table are skipped — re-running `npm run fetch` never re-scores existing articles or charges duplicate API costs; new articles fetched in subsequent runs are picked up automatically

## [0.0.2] - 2026-04-14
### Added
- **Parallel RSS scraping** (ADD 1): `fetchFeed()` wraps `rss-parser` with a 10-second timeout per feed; `runFetch()` launches all enabled feeds via `Promise.allSettled` — one feed returning 4xx/timeout does not block or crash the rest; failed sources logged at warn with reason; `FetchResult` reports per-source success/failure counts
- **URL dedup via `INSERT OR IGNORE`** (ADD 2): `articles` table has `UNIQUE` constraint on `url`; `insertArticle()` uses `INSERT OR IGNORE` — second fetch of the same article silently skips; returns `{ inserted: boolean, id }` so caller knows whether to proceed with OG extraction and state creation; fetch summary shows `Duplicates: N` count
- **OG image extraction** (ADD 3): `og-extractor.ts` fetches article HTML with a 5-second timeout, regex-extracts `og:image` meta tag content; max 5 concurrent requests (via manual semaphore) to avoid overwhelming source sites; never throws — on any error returns `null`; `updateOgImage()` called only on successful extraction; missing OG image is not a failure condition
- **Article state machine** (ADD 4): `article_states` table tracks one row per article; states: `new` → `read` → `starred` → `drafted` → `posted` → `dismissed`; `createArticleState(id, 'new')` called on first insert; `updateArticleState()` used by scoring (auto-dismiss) and future CLI state transitions; `getArticleState()` used in list queries
- **Source upsert** (ADD 5): `seedSources()` uses `INSERT INTO sources ... ON CONFLICT(slug) DO UPDATE SET url=..., enabled=..., tier=...` — editing `rss-sources.ts` (URL, enabled flag, tier) auto-syncs to DB on next fetch without manual migration; `getEnabledSources()` filters by `enabled = 1`; `updateLastFetchedAt()` stamped after each successful feed fetch
- **CLI list with tiered output** (ADD 6): `npm run list` queries `articles JOIN article_states JOIN filter_results JOIN sources` with flexible WHERE; default view shows HOT then OTHER sections separated by a divider; `--all` overrides tier filter; `--limit=N` caps results; prints article ID (for `brief` command), score, title truncated to 50 chars, source name, and time-ago; footer shows counts per tier and active recency window
- **39 Vitest tests** (ADD 7): `tests/database.test.ts` (8 tests: schema init, posts CRUD for platform `x`), `tests/feed-fetcher.test.ts` (14 tests: source seeding, article dedup, article states, mocked `runFetch`), `tests/content-filter.test.ts` (10 tests: filter DB CRUD, scoring + state transitions, mocked Anthropic, fallback score, cost calc), `tests/research-briefer.test.ts` (7 tests: brief cache, related articles query, mocked Sonnet, API failure fallback, user context fallback, cost calc); all use in-memory SQLite via `resetDb()`

## [0.0.1] - 2026-04-12
### Added
- **SQLite schema — 7 tables** (ADD 1): `sources` (slug UNIQUE, name, url, category, tier, language, enabled, last_fetched_at), `articles` (url UNIQUE, title, content, summary, author, published_at, og_image_url, source_id FK), `article_states` (article_id UNIQUE FK, state, updated_at), `filter_results` (article_id UNIQUE FK, score, category, reasoning, suggested_angle, ai_context, tokens_in, tokens_out, model), `drafts` (planned — Slice 5), `posts` (planned — Slice 5), `post_metrics` (planned — Slice 5+); WAL mode + foreign keys enabled at connection init; `migrations.ts` detects old table shapes and drops/recreates — safe because sources are re-seeded and scoring is idempotent
- **Singleton DB connection** (ADD 2): `initDb()` / `getDb()` / `closeDb()` / `resetDb()` in `database/index.ts`; `getDb()` throws if called before `initDb()` — prevents accidental use before setup; `resetDb()` used by tests to get a fresh in-memory DB per test file; `DB_PATH` resolves `~` prefix via `os.homedir()` — defaults to `~/.inkpilot/inkpilot.db`
- **Posts CRUD** (ADD 3): `insertPost()` inserts to `posts` table with `platform` field (`'x'` for Twitter); `getPostById()`, `getPostsByPlatform()`, `countTodayPosts()` for future publishing adapter (Slice 5); `post_metrics` table column named `recasts` — will be renamed to `reposts` in Slice 5 when X API is wired up
